# Практика #3

## Если есть иерархия байтовых потоков ввода-вывода, основанная на абстрактных классах InputStream и OutputStream, то зачем дополнительные иерархии, основанные на абстрактных классах Reader и Writer?

Работа с текстом (символами) и работа с байтами — это разные задачи.


```InputStream/OutputStream``` работают с байтами (8 бит), а ```Reader/Writer``` — с символами (16 бит в Unicode).

Один символ может занимать 1, 2, 3 или даже 4 байта в зависимости от кодировки (UTF-8, UTF-16, Windows-1251 и т.д.). ```InputStream``` читает байты и не знает, где границы символов. ```Reader``` автоматически обрабатывает кодировки и собирает байты в корректные символы.


Байтовые потоки (InputStream/OutputStream) — для бинарных данных (изображения, видео, аудио, архивы)

Символьные потоки (Reader/Writer) — для текстовых данных (файлы с текстом, XML, JSON, HTML)

# Практика #4

## Зачем нужен интерфейс AutoCloseable? Приведите пример.

Он нужен для освобождения ресурсов (закрытие файлов, соединений с БД, сетевых сокетов).

Раньше программисты должны были вручную закрывать ресурсы в блоке ```finally```, что приводило к громоздкому коду и утечке памяти/ресурсов.

Интерфейс ```AutoCloseable``` имеет единственный метод ```close()```. В сочетании с конструкцией ```try-with-resources``` ресурсы закрываются автоматически.

Пример:

```java
// Без AutoCloseable (старый способ)
FileInputStream fis = null;
try {
    fis = new FileInputStream("file.txt");
    // работа с файлом
} finally {
    if (fis != null) fis.close(); // не забыть
}

// С AutoCloseable (новый способ)
try (FileInputStream fis = new FileInputStream("file.txt")) {
    // работа с файлом
} // fis.close() вызовется автоматически
```
